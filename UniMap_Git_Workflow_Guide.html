0) Ce ai deja

✅ Repo GitHub: cosmintrica/UniMap   •   ✅ SSH configurat • ✅ Primul push reușit.

Acest ghid conține toate comenzile de care ai nevoie pentru a lucra la proiect. Urmează pașii și vei fi în regulă!

1) Concept — Fluxul Recomandat (cu Branch-uri)

Gândește-te la proiect ca la o carte. Nu scrii direct în varianta finală, ci lucrezi pe ciorne.

  • main: Este varianta oficială, tipărită a cărții. Mereu trebuie să fie stabilă și funcțională. Nimeni nu lucrează direct aici.
  • Feature Branch (ex: feat/login-ui): Este o ciornă unde adaugi un capitol nou sau corectezi greșeli. Orice modificare se face pe o astfel de "ciornă".
  • Pull Request (PR): Este cererea ta de a adăuga "ciorna" la cartea oficială. Aici, alții pot revizui munca ta înainte de a o accepta.
Acest flux este standardul în industrie deoarece este sigur, organizat și esențial pentru colaborare.

2) Flux Alternativ: Push Direct în main (Proiecte Solo)

Acesta este un flux simplificat în care sari peste pașii de branch și Pull Request. Lucrezi direct pe main și trimiți modificările direct pe server.

Pașii pentru Push Direct:

# 1. Asigură-te că ești pe main și ai cea mai nouă versiune
git switch main
git pull

# --- Aici lucrezi în Xcode ---

# 2. Pregătește fișierele modificate
git add -A

# 3. Salvează modificările cu un commit
git commit -m "fix: am corectat o eroare de afișare"

# 4. Trimite commit-ul direct pe server în ramura main
git push
Atenție! Când să folosești acest flux:
  • ✅ Doar dacă lucrezi singur la proiect.
  • ✅ Pentru modificări foarte mici și sigure (ex: corectarea unei greșeli de text).
De ce este riscant și nerecomandat în echipe:
  • ❌ Fără plasă de siguranță: Orice greșeală (bug) ajunge direct în versiunea oficială și poate strica aplicația.
  • ❌ Fără Code Review: Nimeni nu îți poate verifica munca înainte de a fi integrată, crescând riscul de erori.
  • ❌ Istoric murdar: Istoricul ramurei main se umple de commit-uri mici, greu de urmărit.

3) Pașii de lucru (cu Branch-uri)

Acesta este ciclul pe care îl vei repeta pentru fiecare funcționalitate nouă. Deschide un Terminal în folderul proiectului (acolo unde e UniMap.xcodeproj).

Pasul A: Sincronizează și creează branch-ul

# 1. Mergi pe ramura principală (cartea oficială)
git switch main
# 2. Descarcă cele mai noi modificări de pe server
git pull
# 3. Creează (-c) o "ciornă" nouă și treci pe ea
git switch -c feat/onboarding-polish

Pasul B: Lucrează și salvează progresul (Commit)

Acum poți lucra în Xcode. Când ai un set de modificări logice pe care vrei să le salvezi, rulează următoarele comenzi:

# 1. Pregătește TOATE fișierele modificate (-A) pentru "fotografie"
git add -A
# 2. Fă "fotografia" (commit) și adaugă un mesaj (-m) descriptiv
git commit -m "polish: onboarding UI + compact reset panel"

Pasul C: Trimite munca pe GitHub (Push)

Când ești gata să trimiți "ciorna" pe server pentru a crea un Pull Request:

# Încarcă branch-ul tău pe serverul GitHub (numit 'origin')
git push -u origin feat/onboarding-polish

4) Pull Request & Merge în main

Odată ce ai deschis linkul de Pull Request (PR) pe GitHub:

  1. Completezi un titlu și o descriere clară pentru PR (ce face, cum se testează).
  2. Verifici dacă trec toate verificările automate (dacă ai configurat GitHub Actions).
  3. Apasă butonul verde **"Merge pull request"**. Se recomandă opțiunea **"Squash and merge"**.
  4. După merge, apasă butonul **"Delete branch"** pentru a șterge "ciorna" de pe server.

Curățenie locală

Acum că munca ta e în `main`, trebuie să-ți aduci la zi folderul local.

# 1. Mergi înapoi pe ramura principală
git switch main
# 2. Descarcă noua versiune de `main` care conține și munca ta
git pull
# 3. Șterge (-d) "ciorna" locală, nu mai ai nevoie de ea
git branch -d feat/onboarding-polish

5) Protejează ramura main

Pentru a preveni push-urile directe accidentale în `main`, poți adăuga o regulă de protecție. Este foarte recomandat.

Pe GitHub: Repository → Settings → Branches → Add branch protection rule:

  • Branch name pattern: main
  • Bifează: Require a pull request before merging
  • (Opțional) Bifează: Require status checks to pass before merging — dacă ai workflow de build, acest lucru asigură că doar codul care compilează cu succes poate fi integrat.

6) Invită colaboratori

Dacă vrei să lucrezi cu altcineva la proiect:

Repo → Settings → Collaborators → Add people → introdu username‑ul persoanei → alege permisiunea (de obicei `Write`).

7) README & CONTRIBUTING

Fișierele `README.md` și `CONTRIBUTING.md` sunt esențiale pentru a documenta proiectul și regulile de contribuție.

cat > README.md << 'EOF'
# UniMap

Hărți interioare pentru universități (Craiova → prima).
**Stack:** SwiftUI, MapKit, JSON pentru hărți, editor intern.

## Rulare
1. Deschide `UniMap.xcodeproj`
2. Alege schema: UniMap
3. Rulează (⌘R)

## Structură
- App/ — entry point
- Features/ — ecrane (funcționalități)
- Shared/ — componente comune (reutilizabile)
- Data/Maps — hărțile în format JSON
EOF

cat > CONTRIBUTING.md << 'EOF'
## Flux de lucru
1) `git switch main` + `git pull`
2) `git switch -c feat/nume-feature`
3) `git add -A` + `git commit -m "feat: ..."` + `git push -u origin feat/...`
4) Creează Pull Request → „Squash and merge”
5) `git switch main` + `git pull` + `git branch -d feat/...`
EOF

git add README.md CONTRIBUTING.md
git commit -m "docs: add README & CONTRIBUTING"
git push

8) Git LFS pentru fișiere mari

Git nu este eficient la stocarea fișierelor binare mari (imagini, PDF-uri). Git LFS (Large File Storage) rezolvă asta.

# Instalează extensia LFS (o singură dată per computer)
brew install git-lfs

# Activează LFS pentru proiectul curent (o singură dată per proiect)
git lfs install

# Spune-i lui Git LFS ce tipuri de fișiere să urmărească
git lfs track "*.png" "*.jpg" "*.jpeg" "*.pdf"

# Adaugă fișierul de configurare .gitattributes la commit
git add .gitattributes
git commit -m "chore: enable Git LFS"
git push
După acest pas, poți adăuga fișiere mari în proiect, iar Git le va gestiona eficient.

9) Build automat (GitHub Actions)

Poți configura GitHub să compileze automat proiectul la fiecare Pull Request sau push în `main` pentru a te asigura că nu ai introdus erori.

mkdir -p .github/workflows
cat > .github/workflows/ios-build.yml << 'EOF'
name: iOS Build & Test

on:
  pull_request:
    branches: [ main ]
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: macos-14
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        
      - name: Select Xcode version
        run: sudo xcode-select -s /Applications/Xcode.app
        
      - name: Resolve Swift Package Manager dependencies
        run: xcodebuild -resolvePackageDependencies -project UniMap.xcodeproj
        
      - name: Build for iOS Simulator
        run: |
          xcodebuild -project UniMap.xcodeproj \
                     -scheme UniMap \
                     -destination 'platform=iOS Simulator,name=iPhone 15' \
                     build | xcpretty
        shell: bash
EOF

git add .github/workflows/ios-build.yml
git commit -m "ci: add iOS build workflow"
git push

10) Probleme frecvente

„Updates were rejected because the remote contains work that you do not have locally.”

Asta înseamnă că altcineva (sau tu, de pe alt computer) a făcut un `push` pe server. Trebuie să descarci întâi acele schimbări.

git pull --rebase
git push

Nu te lasă să schimbi branch-ul pentru că ai modificări ne-comise.

# Opțiunea 1: Salvezi temporar modificările
git stash
# ...schimbi branch-ul, faci ce ai de făcut...
git switch main
# ...apoi aplici înapoi modificările salvate
git stash pop

# Opțiunea 2: Faci un commit provizoriu
git add -A
git commit -m "WIP" # WIP = Work In Progress

Remote-ul este pe HTTPS și cere mereu user/parolă? Treci pe SSH.

git remote set-url origin git@github.com:cosmintrica/UniMap.git
# Verifică dacă s-a schimbat
git remote -v

11) Xcode GUI (alternativ)

Dacă preferi o interfață grafică în locul Terminalului, Xcode oferă suport de bază pentru Git.

  • Commit/Push: Meniul de sus → Source Control → Commit… → bifează "Push to remote".
  • Pull: Meniul de sus → Source Control → Pull…
  • Branch nou: În panoul din stânga (Project Navigator), click dreapta pe folderul proiectului → New Branch…
  • Schimbă branch: În meniul de sus → Source Control → Switch to Branch…
  • Pull Request: Pentru asta trebuie să mergi pe site-ul GitHub. După ce faci `Push` la un branch nou, GitHub îți va afișa un buton mare, galben, pentru a crea un PR.

Cheat sheet ultra-scurt

# Fluxul recomandat (cu branch-uri)
git switch main && git pull
git switch -c feat/nume
# ...lucrezi...
git add -A
git commit -m "feat: schimbare"
git push -u origin feat/nume
# ...faci merge pe GitHub...
git switch main && git pull
git branch -d feat/nume
